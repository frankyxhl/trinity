## What

Add a new Trinity CLI subcommand `trinity session-path <provider>[:<instance>] [--project <dir>]` that resolves a session pointer (provider+instance → `session_id` from `<project>/.claude/trinity.json`) to the **absolute path** of that session's on-disk JSONL transcript, dispatches per-provider to a layout-aware resolver, verifies file existence, and prints one path line to stdout. Path-only contract: no transcript content is read, parsed, or surfaced. Errors go to stderr with a one-line message.

Concretely:

- **New module `scripts/session_path.py`** (sibling to `scripts/session.py`, NOT additions to `session.py`). Recommended location based on the trade-off table below; plan-review may override and fold the resolvers into `session.py`.
- **CLI dispatcher wiring in `scripts/codex.py`** — new `session-path` subparser registered via `subparsers.add_parser("session-path")` in `build_parser()` (around L2605, alongside existing `init-config` / `doctor` / `review` / `status` subparsers); new `cmd_session_path(args)` function dispatched from `main()`. Reserved-words set `PRESET_ALIAS_RESERVED_WORDS` (codex.py:32) gains `"session-path"` to prevent preset-alias collision (mirrors how existing `"status"` / `"doctor"` / `"review"` / `"help"` / `"init-config"` are reserved).
- **Per-provider resolvers** — five concrete resolver functions plus one stub:
  - `_resolve_glm(session_id, project_dir)` → `~/.factory/sessions/<encoded-project-path>/<session-id>.jsonl`. Encoding scheme: replace `/` with `-` on the absolute project path (matches droid's existing layout per `~/.factory/sessions/-Users-frank-Projects-trinity` etc.).
  - `_resolve_codex(session_id, project_dir)` → date-bucketed `~/.codex/sessions/<YYYY>/<MM>/<DD>/<session-id>.jsonl`. Primary lookup via `~/.codex/session_index.jsonl` (line-delimited JSON; one record per session with the date keys); fallback glob `~/.codex/sessions/*/*/*/<session-id>.jsonl` if the index entry is absent or unreadable. Index-first reduces filesystem scan cost in the common path; glob fallback preserves correctness when the index is stale or missing.
  - `_resolve_claude_family(session_id, project_dir)` → `~/.claude/projects/<encoded-project-path>/<session-id>.jsonl`. Used by all three claude-code-family providers: `claude-code`, `deepseek`, `openrouter` (matches their shared wrapper layout). Encoding scheme: same `/` → `-` replacement on the absolute project path.
  - `_resolve_gemini(session_id, project_dir)` → STUB. Exits 3 with stderr message `gemini transcript layout not yet supported (TRN-3040)`. No filesystem probe attempted — explicit deferral, not silent failure (per AC bullet "explicit, not silent").
- **Provider→resolver dispatch table** inside `session_path.py` — small Python dict literal mapping provider name (`"glm"`, `"codex"`, `"claude-code"`, `"deepseek"`, `"openrouter"`, `"gemini"`) to the resolver callable. Unknown provider → exit 3 with stderr `provider 'X' not supported by session-path resolver`. Dispatch table is the single source of truth for "which providers support session-path"; `providers/registry.json` is NOT modified in v1 (the registry's role is CLI invocation; the resolver lookup is path-resolution layer, not provider invocation). v2 may unify under `transcript_path_template` field in the registry — out of scope here.
- **Tests `tests/test_session_path.py`** — at minimum one test per resolver (5 happy-path tests: glm, codex via index, codex via glob fallback, claude-family ×3 collapsed into one parameterized test or three short tests, gemini stub returns exit 3); plus three error-exit tests (no pointer = exit 2, pointer-but-file-missing = exit 3, unknown-provider = exit 3); plus one path-only-contract assertion (resolver functions must not call `open()` / `read()` on the resolved path — guarded via `unittest.mock.patch` on builtins or via grepping the implementation). Tests use `tmp_path` fixtures with fake `.claude/trinity.json` pointers and fake empty JSONL files at the expected on-disk locations under a `tmp_path`-rooted fake `$HOME` (set via `monkeypatch.setenv("HOME", str(tmp_home))`).
- **Documentation update** — one short section in `SKILL.md` (the active skill-document at `/Users/frank/Projects/trinity/SKILL.md`, 415 lines, used by Claude/agents at session start) titled `## Session Transcript Paths`. Includes the command syntax, a per-provider transcript-root table, and the three exit codes. README.md update is NOT REQUIRED — SKILL.md is the actionable surface for agents and is where the `session.py --version` reference already lives (SKILL.md:28). Plan-review may override to ALSO update README if the user-facing docs need a mention.

Implementer extracts during phase 5; both modules import from the shared helper. |

If a future evolve cycle finds the fan-out cost is real, that's a separate CHG. The clause at the top of this section that referred to plan-review-override under CLD-1802 §2 surface taxonomy is superseded by this lock.

Per CLD-1802 §2 surface taxonomy — both options touch one new module + the dispatcher; (A) is chosen for separation-of-concerns and aligns with TRN-1800's atomicity weight (one module = one purpose). See the locked decision below.

## Why

Today, finding the JSONL transcript for a Trinity session is manual: read `.claude/trinity.json` to get the `session_id`, then remember the per-provider directory layout, then glob/grep. There is no Trinity-level command for this, which puts a friction wall in front of:

- **Token-efficiency audits** — the user explicitly raised this in the originating chat. Comparing prompt sizes, response sizes, and round-counts across providers requires reading JSONL, but step 0 is "know where the JSONL is".
- **Prompt-bloat investigations** — when an agent's prompts grow over a session, replay debugging needs the transcript file path.
- **Cross-provider replay debugging** — currently every provider's transcript root is in a different mental drawer. One command unifies the surface.

The alternative (status quo) is each downstream consumer re-implementing the same per-provider layout knowledge. Path-only is deliberately the smallest viable surface — it does NOT read transcripts, normalize schemas, or count tokens. Those are downstream consumer work that pipes to `cat`, `jq`, an LLM, or a future `trinity session-log` reader (out of scope per #108).

## Threat Model assessment

The `session-path` command has a deliberately narrow attack surface — it reads ONLY the pointer file (`.claude/trinity.json`) and probes for file existence (no content reads). Specifically:

- **No transcript content read**: stdout output is a single absolute path string; the resolver never opens, reads, parses, or surfaces any portion of the JSONL transcript. Acceptance Criteria includes a grep assertion to enforce this contract. JSONL transcripts may contain sensitive prompt/response history; not reading them keeps PII exposure flat versus today.
- **Pointer-file integrity**: the resolver reads `.claude/trinity.json` via the same `fcntl.flock(LOCK_SH)` shared-lock pattern `scripts/session.py:cmd_read` already uses (codex/glm/etc. providers may write the pointer concurrently from their own session lifecycle). No write-lock is taken — read-only path.
- **Path traversal**: `session_id` values in the pointer file are written by `session.py:cmd_write` and originate from provider CLIs (droid / codex / claude wrappers). These are UUIDs / opaque tokens, not user input — but as defense-in-depth, the resolver MUST validate `session_id` matches the conservative regex `^[a-zA-Z0-9_\-.]+$` (canonical regex, identical to Surface 1 + AC line 84; allows `.` because some legacy session IDs contain it; rejects `/`, `..` sequences via the absence of `/`, whitespace, null bytes) before joining into a filesystem path. A malicious or corrupt pointer entry containing `..`, `/`, or whitespace MUST cause exit 3 with stderr `invalid session_id format`, not a filesystem traversal. This guard is REQUIRED, not advisory — Surface 1 stderr catalog lists the message verbatim and AC line 84 enforces the rejection path with a dedicated test bullet.
- **Stderr message hygiene**: per #108 task design, stderr messages are short and structured. The resolver MUST NOT echo full search globs (e.g., do NOT print `tried glob '~/.codex/sessions/*/*/*/<id>.jsonl'`); use the structured strings listed in Surface 1 instead. Rationale: error messages are an information-disclosure surface — leaking the per-provider search layout to a low-trust caller (e.g., a wrapper that pipes stderr to a remote log) reveals filesystem structure that the path-only contract intends to keep implicit. Concrete strings: `gemini transcript layout not yet supported (TRN-3040)` (no glob); `provider 'X' not supported by session-path resolver` (no list of supported providers); `transcript file not found at '<path>'` (the resolved path, which is what the caller is asking for, IS surfaced — that is the intended output even on failure); `no session pointer for '<key>'` (echoes the caller-supplied key, no filesystem disclosure); `invalid session_id format` (no echo of the malformed value, to prevent log-injection).
- **No new attack vectors vs. existing `session.py`**: `session.py` already reads `.claude/trinity.json`. The new command extends that read with a `os.path.isfile` probe on a file path constructed from `session_id` + per-provider layout. File-existence probes do not reveal contents; even on a multi-user system, JSONLs under `~/.factory/`, `~/.codex/`, `~/.claude/projects/` are scoped to the invoking user's home directory.
- **Symlink attack**: `os.path.isfile` follows symlinks. If an attacker can write to `~/.factory/sessions/.../<session-id>.jsonl` (which would require write access to the user's home — a higher privilege than running `trinity session-path`), they can redirect the resolved path. The output is still just a path string; the caller decides whether to read it. Since the caller (user / downstream tool) is the one with read intent, the trust boundary is unchanged. Accepted residual.
- **Concurrent pointer-file writes**: `session.py:cmd_write` takes `fcntl.flock(LOCK_EX)` exclusive lock; the `session-path` reader takes `LOCK_SH` shared lock. Standard reader-writer pattern; no torn reads.

**Accepted residuals**: (a) symlink-following on `os.path.isfile` (mitigation requires the caller to control its own home directory, which is the existing trust assumption); (b) caller may chain `cat $(trinity session-path glm)` to read the transcript — that is the design intent of path-only output, and the caller's read decision; (c) date-bucketed codex glob fallback may match multiple files if `session_id` collides across days (probability negligible — UUIDs); the resolver MUST exit 3 with `multiple transcript files matched session_id '<id>'` if `len(glob_matches) > 1`, not silently pick one.
