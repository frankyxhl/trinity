## What

Add a `minimax` provider that mirrors the existing `glm` wiring — same `droid exec` CLI shape, same session-management code path, same structured-review (TRN-3022) contract — routed to MiniMax 2.7. Source-authored file: `providers/minimax.delta.md` (mirrors `providers/glm.delta.md` with model/brand/instance-key swaps). Build output: `providers/minimax.md` — generated by `make build`, verified by `make verify-built` (CI determinism gate). Registry, install paths, tests, and docs all gain a `minimax` entry.

**Placement convention:** All `minimax` insertions go **after `glm`** in alphabetical/grouping conventions — droid-backed providers grouped (glm, minimax), wrappers follow (codex, openrouter, deepseek, claude-code). Gemini stays excluded per TRN-3020 GEMINI_EXCEPTION.

## Why

Trinity ships six providers. Only `glm` uses Factory's `droid` CLI; MiniMax 2.7 is now also reachable via `droid exec --model minimax-m2.7`. Adding it gives the orchestrator a non-Anthropic, non-OpenAI voice for review panels and `/trinity` dispatch, reducing GLM's single-point-of-failure for `droid`-backed paths. Zero new wrapper script — GLM session pattern reused verbatim. Full spec: issue #103.

## MiniMax 2.7 model identifier

Verified at CHG-draft time via `droid exec --model help` (an invalid id triggers droid to print the available-models list): the built-in id `minimax-m2.7` is present, alongside a custom variant `custom:MiniMax-M2.7-3`. The CHG fixes the registry CLI to the **built-in id `minimax-m2.7`** to avoid coupling to per-machine `~/.factory/settings.json` custom-model definitions. If at Phase 5 the implementer's runtime no longer lists `minimax-m2.7` (droid model catalog drift), the CHG must be reworked or the issue closed as not-yet-feasible — this is a runtime dependency, not a plan-review decision-deferral.

## Migration / Backward-compat

Zero impact on existing providers — purely additive. `providers/registry.json` gains one key (no existing key changed). ISSUE_TEMPLATE dropdown gains one option (existing options unchanged). No public API removal, no config migration, no release-note breaking change. Review-panel composition (TRN-1008 §4 / §8) is **not** changed by this CHG; `minimax` becomes available for opt-in `/trinity` dispatch only. Panel inclusion is a separate decision tracked elsewhere (see issue #88).

**Out of scope:** `.agents/trinity.codex.json` is intentionally NOT updated — codex-side test_a3b uses overlap intersection (set & set), not set-equality, so adding a new provider to registry without codex.json doesn't break it. Codex-side opt-in tracked separately.

**SKILL.md install table (L329-336):** minimax is reachable via the existing `pip install factory-droid` row (same droid CLI as glm); no new install-spec row needed.

**Compression:** N/A — additive feature; no deletion offset possible. The CHG itself is structurally compressed (each surface ≤ 1 line; AC mirrors surfaces).
